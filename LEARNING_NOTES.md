# 학습 노트 1 — vec3부터 hittable까지

RTIOW CPU 버전 짜면서 "코드는 됐는데 왜 되는지" 걸린 것들 정리.
그래픽스 알고리즘 자체보다, 그걸 C++로 옮길 때 걸린 **언어·저수준 개념**이 대부분이었음.
(코테 위주로 C++ 써와서 클래스 설계·메모리·연산자 오버로딩이 약했는데, 이 프로젝트로 메움.)

## 1. vec3 — 연산자 오버로딩

### 이항 연산자를 왜 클래스 밖 자유 함수로 두나
멤버 연산자로 쓰면 `a + b`가 `a.operator+(b)`로 해석돼서 **왼쪽 피연산자가 반드시 그 클래스 객체**여야 함.
근데 `2.0 * v`는 왼쪽이 `double`. 내장 타입엔 멤버 함수 못 붙임 →
**스칼라가 왼쪽 오는 연산은 자유 함수 말곤 자리가 없음.**
그래서 이항 산술(`+ - * /`)은 통일성 위해 다 자유 함수로. `v * 2.0`은 `2.0 * v`로 떠넘기면 교환편리.

```cpp
inline vec3 operator*(double t, const vec3& v);                  // 스칼라 * 벡터
inline vec3 operator*(const vec3& v, double t) { return t * v; }  // 떠넘김
```

### const vec3& v — const가 뭘 보장하나
- `vec3&` = 복사 말고 원본을 참조로 받아라 (24바이트 복사 회피)
- `const` = 받되 함수 안에서 원본 안 바꾸겠다는 약속

→ "복사도 안 하고 원본도 안 건드림". 읽기만 하는 인자의 표준형.

### return *this — 계산 반영 시점 (한참 헷갈렸음)
```cpp
vec3& operator+=(const vec3& v) {
    e[0] += v.e[0]; ...   // ← 여기서 this(=a)가 실제로 바뀜. 계산 반영 완료.
    return *this;         // ← 이미 바뀐 a를 내보냄 (연쇄 쓰라고)
}
```
- `const` 붙은 건 **인자 v(=b)뿐**. v는 안 바뀜.
- `this`(=a)엔 const 없어서 바뀜. **계산은 return이 아니라 `+=` 세 줄에서 이미 끝남.**
- `return *this`는 바뀐 결과를 다시 쓰게 넘기는 것. 반영하는 게 아님.
- `this` = 객체 주소(포인터), `*this` = 원본 객체 자신. 참조(`vec3&`)로 반환해서 복사 없이 원본 나감.

### 멤버 함수 뒤 const (double x() const)
"이 함수는 자기 객체(this) 안 바꿈, 읽기 전용"이란 뜻. 숨은 this 포인터를 const로 만드는 것.
실무 이유: **const 객체는 const 멤버 함수만 호출 가능**. 연산자들이 인자를 `const vec3&`로 받으니
그 안에서 `u.x()` 부르려면 `x()`가 const여야 함. → **읽기만 하면 뒤에 const 붙이기.**

### inline — 성능이 아니라 ODR 때문
정의를 헤더에 두면, 그 헤더를 여러 `.cpp`가 include할 때 링커가 같은 정의를 여러 번 봐서
**ODR 위반** 링크 에러. `inline` = "중복돼도 같은 놈이니 합치고 에러 내지 마".
헤더 온리에서 inline의 핵심 의미는 "인라인 전개(성능)"가 아니라 **중복 정의 허용(링킹)**.
(클래스 본문 안 멤버 함수는 자동 inline이라 키워드 필요 없음.)

### 멤버 이니셜라이저 리스트가 왜 "바로 초기화"
클래스 안 `double e[3];`은 실행 문장이 아니라 **설계도(선언)**. 실제 메모리 확보+초기화는 객체 만들 때(`vec3 a;`) 한 번에.
- 이니셜라이저 리스트: 확보와 동시에 `{0,0,0}` 생성 → **쓰레기값 단계 없음**
- 본문 대입(`{ e[0]=0; ... }`): 확보 → 쓰레기값 → 나중에 덮어씀 (한 단계 더)

double은 어차피 덮어써서 결과 같지만, **const 멤버·참조 멤버·기본 생성자 없는 멤버는 이니셜라이저가 유일한 방법.**

### cross — 조용한 버그 1순위
외적은 성분 순서·부호 틀려도 **컴파일 되고 그림만 미묘하게 틀어짐.**
공식과 한 성분씩 대조하고, `cross((1,0,0),(0,1,0)) == (0,0,1)`처럼 **손으로 예상값 먼저 구해** 검증.
(테스트 통과 ≠ 정답. 예상값 먼저 정해야 함.)

## 2. 접근 지정자 (public/private)

- `public`: 밖에서 접근 가능 / `private`: 클래스 안에서만 / `protected`: 자식까지
- 관례: 변수는 private(캡슐화), 함수는 public(창구) — **지킬 불변조건 있는** 클래스일 때
- 예외: vec3처럼 지킬 규칙 없는 순수 데이터 묶음은 다 public (RTIOW 스타일)
- 순수 컴파일 타임 검사. 런타임 비용 0, 메모리 레이아웃 영향 없음.
- class vs struct 차이는 기본 접근 수준(private vs public)뿐.

## 3. ray

### 멤버 선언 순서
멤버는 **선언 순서대로** 초기화됨. 이니셜라이저에 뭐라 쓰든 무관.
한 멤버가 다른 멤버 참조해 초기화되면 순서 때문에 조용한 버그.
→ **선언 순서 = 이니셜라이저 순서** 맞추기 (`-Wreorder`가 경고).

```cpp
class ray {
  public:
    point3 orig;   // 먼저 선언 → 먼저 초기화
    vec3 dir;
    ray(const point3& origin, const vec3& direction)
        : orig{origin}, dir{direction} {}   // 멤버{값} 짝 맞춤
};
```

### origin()이 값 대신 참조(const point3&) 반환하는 이유
`orig`는 이미 객체 안에 살아있는 멤버라 복사 말고 참조로 주면 됨.
반면 `operator+`는 **새 벡터를 만들어** 반환하니 참조 주면 dangling(지역 변수 소멸).
→ 기준: **반환 대상이 호출 후에도 살아있으면 참조, 함수 안에서 새로 만든 거면 값.**

`at()`이 처음으로 내가 만든 vec3 연산자들이 조립되는 지점:
```cpp
vec3 at(double t) const { return orig + t * dir; }
// t * dir → operator*(double, const vec3&)
// orig + (…) → operator+(const vec3&, const vec3&)
```

## 4. 광선-구 교차

### "부딪혔나?"가 왜 판별식 하나로 환원되나
점 P가 중심 C, 반지름 r 구 표면 위 조건은 `(P−C)·(P−C) = r²`.
P를 광선 `P(t) = A + t·b`로 치환하면 **t에 대한 2차방정식**:
```
(b·b) t² + 2(b·(A−C)) t + ((A−C)·(A−C) − r²) = 0
```
판별식 `D = B² − 4AC` 부호가 근의 개수 = 교차 개수:
- `D > 0` → 근 2개 → 관통(앞면·뒷면)
- `D = 0` → 접함
- `D < 0` → 안 맞음

기하 문제(부딪혔나)가 대수 판정(D ≥ 0)으로 환원되는 게 레이트레이싱의 우아한 지점.

### 근 두 개일 때 왜 (−b−√D) 작은 t를 고르나
`a = dot(dir,dir)`가 양수라 분모 `2a > 0` → √D 빼는 쪽이 항상 작은 t.
작은 t가 카메라에서 **먼저 만나는 지점(가까운 앞면)**. 보이는 건 가까운 표면이고 뒷면은 가려짐.
→ **가장 가까운 교차점을 그린다** (나중에 z-buffer의 레이트레이싱 버전으로 확장).

### 카메라 뒤(t<0) 구가 왜 보였나 (당시 못 걸른 한계)
판별식만 보면 "무한 직선이 구를 지나가나" 판정이라 t 부호를 안 봄.
그래서 `z = +1`(카메라 뒤) 구도 "맞았다"고 판정돼 같은 크기로 보였음.
버그가 아니라 **t ≥ 0 체크를 아직 안 넣은** 상태의 정확한 동작.
→ hit_record + t 범위(`t_min < t < t_max`)로 정식 해결 (노트 2에서 함).

## 5. 표면 법선 시각화

구 표면 점 P의 법선은 **중심에서 바깥으로**: `N = unit_vector(P − C)`. 구는 이게 편리.
법선은 방향이라 성분이 −1~1인데 색은 0~1이어야 하므로:
```cpp
0.5 * (N + vec3(1,1,1))   // −1~1 → 0~1
```
(배경 lerp에서 방향 y를 0~1로 옮긴 것과 같은 변환.)
x→R, y→G, z→B 매핑돼서 구가 알록달록한 입체로 보임.
**"광선이 표면 맞히고, 그 표면이 어느 방향 보는지 안다"** = 모든 음영·반사·재질의 출발점.

## 6. hittable 추상화

(처음엔 `ray_color`에 `hit_sphere`를 하드코딩했다가, 여러 물체 다루려고 공통 인터페이스로 리팩터링. 실제 구현은 노트 2.)

- `hittable`: 순수 가상 함수 `hit(...) = 0`을 가진 추상 베이스. 객체 생성 불가.
- `sphere`가 상속받아 `hit` 구현. **동적 디스패치(vtable)**로 실제 타입의 hit이 불림.
- `hit`은 `bool`(맞았나)만 반환, 실제 정보(점·법선·t·front_face)는 **`hit_record& rec`을 참조로 받아 채움.**
  → `operator+=`에서 배운 "참조로 원본에 써 넣기" 패턴.
- `virtual ~hittable() = default;` (가상 소멸자): base 포인터로 파생 객체 delete할 때 파생 소멸자가 불리도록.
  → **virtual 함수 있는 base엔 virtual 소멸자** 규칙.
- `front_face`: 광선이 바깥면/안쪽면 어디 맞았나. `dot(dir, outward_normal) < 0`이면 바깥면.
  지금은 안 써도 유리(dielectric)에서 필수.

> **CUDA 북마크**: 이 상속/가상함수/소멸자 구조가 포팅 때 vtable·소유권 문제와 직결.
> CPU선 자연스럽게 두고 "여기 나중에 바뀜"만 표시.
