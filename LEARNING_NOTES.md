## 1. `vec3` — 연산자 오버로딩

### Q. 왜 `operator+`, `operator*`를 클래스 밖 자유 함수로 두는가?

멤버 연산자로 쓰면 `a + b`는 `a.operator+(b)`로 해석되어 **왼쪽 피연산자가 반드시 그 클래스 객체**여야 한다.
그런데 `2.0 * v`는 왼쪽이 `double`이다. 내장 타입에는 멤버 함수를 붙일 수 없으므로
**스칼라가 왼쪽에 오는 연산은 자유 함수 말고는 정의할 자리가 없다.**
그래서 이항 산술(`+ - * /`)은 통일성을 위해 전부 자유 함수로 둔다.
`v * 2.0`은 `2.0 * v`로 떠넘기면 교환법칙이 공짜로 처리된다.

```cpp
inline vec3 operator*(double t, const vec3& v);       // 스칼라 * 벡터
inline vec3 operator*(const vec3& v, double t) { return t * v; }  // 떠넘김
```

### Q. `const vec3& v`에서 `const`는 정확히 뭘 보장하나?

- `vec3&` = 복사하지 말고 원본을 참조로 받아라 (24바이트 복사 회피)
- `const` = 받되 **이 원본을 함수 안에서 바꾸지 않겠다는 약속**

즉 `const vec3&`는 "복사도 안 하고(참조), 원본도 안 건드림(const)" 두 마리를 잡는 조합.
읽기만 하는 함수 인자의 표준 형태다.

### Q. `return *this`에서 계산 결과가 반영되는 시점은?

한동안 헷갈린 지점. `a += b`를 예로 보면:

```cpp
vec3& operator+=(const vec3& v) {
    e[0] += v.e[0]; ...   // ← 여기서 this(=a)가 실제로 바뀜. 계산 반영 완료.
    return *this;         // ← 이미 바뀐 a를 내보냄 (연쇄 사용을 위해)
}
```

- `const`가 붙은 건 **인자 `v`(=b)** 뿐이다. `v`는 안 바뀐다.
- `this`(=a)에는 `const`가 없어서 바뀐다. **계산은 `return`이 아니라 `+=` 세 줄에서 이미 끝났다.**
- `return *this`는 "바뀐 결과를 다시 쓸 수 있게 넘겨주는" 것이지, 반영하는 게 아니다.
- `this`는 객체의 **주소(포인터)**, `*this`는 그 **원본 객체 자신**.
  참조(`vec3&`)로 반환하므로 복사 없이 원본이 나간다.

### Q. 멤버 함수 뒤의 `const`(`double x() const`)는 무슨 뜻?

"이 멤버 함수는 자기 객체(`this`)를 안 바꾼다, 읽기 전용"이라는 뜻.
내부적으로는 숨은 `this` 포인터를 `const`로 만드는 것이다.

이게 실무적으로 중요한 이유: **const 객체는 const 멤버 함수만 호출할 수 있다.**
연산자들이 인자를 `const vec3&`로 받으므로, 그 안에서 `u.x()`를 부르려면 `x()`가 const여야 한다.
→ 규칙: **읽기만 하는 멤버 함수엔 뒤에 `const`를 붙인다.**

### Q. `inline`은 왜 붙이나? (성능 때문이 아님)

정의를 헤더에 두면, 그 헤더를 여러 `.cpp`가 include할 때 링커가 같은 함수 정의를 여러 번 보고
**ODR(One Definition Rule) 위반**으로 링크 에러를 낸다.
`inline`은 링커에게 "이 정의가 중복돼도 같은 놈이니 하나로 합치고 에러 내지 마"라고 말하는 것.
헤더 온리 라이브러리에서 `inline`의 핵심 의미는 "인라인 전개(성능)"가 아니라 **중복 정의 허용(링킹)**이다.
(클래스 본문 안에 정의한 멤버 함수는 자동으로 inline 취급이라 키워드가 필요 없다.)

### Q. 멤버 이니셜라이저 리스트(`: e{0,0,0}`)가 왜 "바로 초기화"인가?

클래스 안의 `double e[3];`은 **실행되는 문장이 아니라 설계도(선언)**다.
실제 메모리 확보와 초기화는 객체를 만들 때(`vec3 a;`) 한 번에 일어난다.

- 이니셜라이저 리스트: 메모리 확보와 동시에 `{0,0,0}`으로 생성 → **쓰레기값 단계 없음**
- 생성자 본문 대입(`{ e[0]=0; ... }`): 확보 → 쓰레기값으로 존재 → 나중에 덮어씀 (한 단계 더)

`double`은 어차피 덮어써서 결과가 같지만, **const 멤버·참조 멤버·기본 생성자 없는 멤버는
이니셜라이저 리스트가 유일한 초기화 방법**이다.

### Q. `cross`는 왜 조용한 버그 1순위인가?

외적은 성분 순서·부호를 틀려도 **컴파일은 되고 그림만 미묘하게 틀어진다.**
그래서 공식과 한 성분씩 대조하고, `cross((1,0,0),(0,1,0)) == (0,0,1)`처럼
**손으로 예상값을 먼저 구해** 검증했다. (테스트 통과 ≠ 정답. 예상값을 먼저 정해야 함.)

## 2. 접근 지정자 (`public`/`private`)

- `public`: 클래스 밖에서 접근 가능 / `private`: 클래스 안에서만 / `protected`: 자식까지
- 관례: 변수는 `private`(캡슐화), 함수는 `public`(창구) — **지켜야 할 불변 조건이 있는** 클래스일 때
- 예외: `vec3`처럼 지킬 규칙이 없는 순수 데이터 묶음은 전부 `public`으로 열어둔다 (RTIOW 스타일)
- 순수 컴파일 타임 검사. 런타임 비용 0, 메모리 레이아웃에 영향 없음.
- `class`와 `struct`의 유일한 차이는 기본 접근 수준(private vs public)뿐이다.

## 3. `ray` — 광선 표현

### Q. 멤버 선언 순서가 왜 중요한가?

C++에서 멤버는 **선언된 순서대로** 초기화된다. 이니셜라이저 리스트에 어떤 순서로 쓰든 무관하다.
한 멤버가 다른 멤버를 참조해 초기화되면 순서 때문에 조용한 버그가 난다.
→ 관례: **선언 순서 = 이니셜라이저 리스트 순서**를 맞춘다 (`-Wreorder`가 경고).

```cpp
class ray {
  public:
    point3 orig;   // 먼저 선언 → 먼저 초기화
    vec3 dir;
    ray(const point3& origin, const vec3& direction)
        : orig{origin}, dir{direction} {}   // 멤버{값} 짝을 맞춤
};
```

### Q. `origin()`이 값(`point3`) 대신 참조(`const point3&`)를 반환하는 이유?

`orig`는 이미 객체 안에 살아있는 멤버라, 복사해서 줄 필요 없이 원본을 참조로 주면 된다.
반면 `operator+`는 **새 벡터를 만들어** 반환하므로 참조로 주면 dangling(지역 변수 소멸)이 된다.
→ 판단 기준: **반환 대상이 호출 후에도 살아있으면 참조, 함수 안에서 새로 만든 것이면 값.**

`at()`은 이 프로젝트에서 처음으로 내가 만든 `vec3` 연산자들이 조립되는 지점이다:

```cpp
vec3 at(double t) const { return orig + t * dir; }
// t * dir → operator*(double, const vec3&)
// orig + (…) → operator+(const vec3&, const vec3&)
```

## 4. 광선-구 교차

### Q. "부딪혔나?"가 왜 판별식 하나로 환원되나?

점 P가 중심 C, 반지름 r인 구 표면 위에 있을 조건은 `(P−C)·(P−C) = r²`.
P를 광선 `P(t) = A + t·b`로 치환하면 **t에 대한 2차방정식**이 나온다:

```
(b·b) t² + 2(b·(A−C)) t + ((A−C)·(A−C) − r²) = 0
```

판별식 `D = B² − 4AC`의 부호가 근의 개수 = 교차 개수를 정한다:

- `D > 0` → 근 2개 → 광선이 구를 관통 (앞면·뒷면)
- `D = 0` → 접함
- `D < 0` → 안 맞음

**기하 문제(부딪혔나)가 대수 판정(D ≥ 0)으로 환원**되는 게 레이트레이싱의 우아한 지점.

### Q. 근이 두 개일 때 왜 `(−b − √D)`(작은 t)를 고르나?

`a = dot(dir, dir)`가 양수라 분모 `2a > 0`이므로, √D를 빼는 쪽이 항상 더 작은 `t`.
작은 `t`가 카메라에서 **먼저 만나는 지점(가까운 앞면)**이다.
우리 눈에 보이는 건 가까운 표면이고 뒷면은 가려지므로 항상 작은 t를 그린다.
→ 레이트레이싱 기본 규칙: **가장 가까운 교차점을 그린다** (나중에 z-buffer의 레이트레이싱 버전으로 확장).

### Q. (당시 못 걸렀던 한계) 카메라 뒤(`t < 0`)의 구가 왜 여전히 보였나?

판별식만 보면 "이 광선을 담은 **무한 직선**이 구를 지나가나"를 판정하는 것이라
`t`의 부호를 따지지 않는다. 그래서 `z = +1`(카메라 뒤)에 놓은 구도 "맞았다"고 판정되어
같은 크기로 보였다. 이건 버그가 아니라 **`t ≥ 0` 체크를 아직 안 넣은** 상태의 정확한 동작.
→ 다음 단계에서 `hit_record` + `t` 범위(`t_min < t < t_max`) 도입으로 정식 해결.

## 5. 표면 법선 시각화

구 표면 점 P의 법선은 **중심에서 바깥으로**: `N = unit_vector(P − C)`. 구는 이게 공짜다.

법선은 방향이라 각 성분이 `−1 ~ 1`인데 색은 `0 ~ 1`이어야 하므로 표준 변환:

```cpp
0.5 * (N + vec3(1,1,1))   // −1~1 → 0~1
```

(배경 lerp에서 방향의 y를 0~1로 옮긴 것과 같은 변환.)
x→R, y→G, z→B로 매핑되어 구가 알록달록한 입체로 보인다.
**"광선이 표면을 맞히고, 그 표면이 어느 방향을 보는지 안다"** — 이게 모든 음영·반사·재질의 출발점.

## 6. (진행 중) `hittable` 추상화 — 예습 정리

`ray_color`에 `hit_sphere`를 하드코딩한 상태에서, 여러 물체를 다루기 위한 공통 인터페이스로 리팩터링 중.

- `hittable`: 순수 가상 함수 `hit(...) = 0`을 가진 추상 베이스. 객체로 생성 불가.
- `sphere`가 이를 상속받아 `hit`을 구현. **동적 디스패치(vtable)** 로 실제 타입의 `hit`이 호출됨.
- `hit`은 `bool`(맞았나)만 반환하고, 실제 정보(점·법선·`t`·`front_face`)는 **`hit_record& rec`를 참조로 받아 채운다.**
  → `operator+=`에서 배운 "참조로 원본에 써 넣기" 패턴과 동일.
- `virtual ~hittable() = default;` (가상 소멸자): base 포인터로 파생 객체를 delete할 때
  파생 소멸자가 불리도록. **virtual 함수가 있는 base엔 virtual 소멸자**가 규칙.
- `front_face`: 광선이 바깥면/안쪽면 중 어디를 맞았는지. `dot(dir, outward_normal) < 0`이면 바깥면.
  지금은 안 써도 유리(dielectric) 재질에서 필수.

> 이 상속/가상함수/소멸자 구조는 **CUDA 포팅 시 vtable·소유권 문제와 직결**되는 지점이라
> 지금은 CPU에서 자연스럽게 두고 "여기 나중에 바뀜"만 표시해 둠 (README의 CUDA 포팅 계획 참고).
